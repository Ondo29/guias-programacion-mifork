En C, donde no existen las excepciones, pongamos un ejemplo de una raíz que toma número flotante positivo. Queremos controlar el error si la función recibe un número negativo. El usuario debe ser informado pero desde fuera de la función raiz 
¿Cómo indicamos ese error?. Enumera dos opciones diferentes de diseñar, poniendo un ejemplo de código de cada una.

Respuesta:

Opción 1: Devolver un valor centinela + usar errno

#include <stdio.h>
#include <math.h>
#include <errno.h>

float raiz(float x) {
    if (x < 0) {
        errno = EDOM;   // Error de dominio matemático
        return -1.0f;   // Valor centinela
    }
    return sqrtf(x);
}

int main() {
    float num = -9.0f;
    float resultado = raiz(num);

    if (errno == EDOM) {
        printf("Error: no se puede calcular la raiz de un numero negativo\n");
    } else {
        printf("Resultado: %f\n", resultado);
    }

    return 0;
}

Opción 2: Devolver código de error y usar parámetro de salida.

#include <stdio.h>
#include <math.h>

int raiz(float x, float *resultado) {
    if (x < 0) {
        return 1;  // Código de error
    }

    *resultado = sqrtf(x);
    return 0;      // Éxito
}

int main() {
    float num = -9.0f;
    float resultado;

    int estado = raiz(num, &resultado);

    if (estado != 0) {
        printf("Error: no se puede calcular la raiz de un numero negativo\n");
    } else {
        printf("Resultado: %f\n", resultado);
    }

    return 0;
}

### 3. Reescribe el mismo ejemplo de raiz, pero en Java, metiendo ese método en una clase Calculadora y llama a dicho método desde el método main, mostrando cómo se puede controlar desde fuera.

Respuesta:

public class Calculadora {

    // Método que puede lanzar una excepción
    public static double raiz(double x) {
        if (x < 0) {
            throw new IllegalArgumentException(
                "No se puede calcular la raíz de un número negativo"
            );
        }
        return Math.sqrt(x);
    }

    public static void main(String[] args) {

        double numero = -9;

        try {
            double resultado = Calculadora.raiz(numero);
            System.out.println("Resultado: " + resultado);

        } catch (IllegalArgumentException e) {
            System.out.println("Error: " + e.getMessage());
        }

        System.out.println("El programa continúa...");
    }
}

### 4. ¿Qué es "lanzar" una excepción? ¿Qué es "controlar" o "capturar" una excepción? ¿Qué es que se "propague" 
una excepción? ¿Qué le va ocurriendo a las funciones en la pila de llamadas por donde se va propagando la excepción? 
¿Las funciones que no la controlan se reanudan después de alguna forma? Explica con el mismo ejemplo anterior en Java 
de la raíz cuadrada.

Respuesta:

¿Qué es lanzar una excepción?

Lanzar una excepción significa:

Crear un objeto de tipo excepción y detener inmediatamente la ejecución normal del método.


¿Qué es capturar o controlar una excepción?

Capturar significa:

Interceptar la excepción con un bloque catch para manejar el error.

¿Qué significa que una excepción se propague?

Propagarse significa:

La excepción sube automáticamente por la pila de llamadas hasta encontrar un catch.


¿Qué ocurre en la pila de llamadas?

Supongamos esta pila.

main
 └── metodoIntermedio
      └── raiz

    Java empieza a “desenrollar la pila” (stack unwinding)

Paso a paso:

Se abandona inmediatamente raiz

Se destruye su contexto (variables locales desaparecen)

Se vuelve a metodoIntermedio

Como no hay catch, también se abandona

Se destruye su contexto

Se llega a main

Allí sí hay catch → se ejecuta

### 11. En Java, qué son las excepciones "controladas" y las "no controladas"? ¿Qué papel juega RuntimeException? Pon un ejemplo de excepciones típicas controladas y no controladas que incluso nosotros mismos podríamos usar. Haz dos listas con 3 o 4 ejemplos 
de situación donde se suele preferir una excepción controlada y donde se suele preferir una excepción no controlada.

Respuesta:

1️⃣ Excepciones controladas (checked)

Se verifican en tiempo de compilación.

El compilador obliga a:

Usar try-catch o declarar throws en el método.

Suelen representar problemas externos previsibles (como fallos de E/S, errores de red, bases de datos).

Clases típicas:

Todas las que extienden Exception pero no RuntimeException.

Ejemplo de declaración y uso:

import java.io.*;

public void leerArchivo(String ruta) throws IOException {
    FileReader fr = new FileReader(ruta); // IOException es checked
}
2️⃣ Excepciones no controladas (unchecked)

No se revisan en tiempo de compilación.

No es obligatorio capturarlas ni declararlas.

Suelen representar errores de programación:

Llamadas inválidas

Valores nulos

Desbordamientos, etc.

Todas las que extienden RuntimeException y Error.

RuntimeException es la clase base de estas excepciones. Permite crear excepciones propias que no requieren try-catch.

Ejemplos comunes:

throw new NullPointerException("objeto nulo");
throw new IllegalArgumentException("argumento inválido");

Listas de situaciones típicas
🟢 Situaciones donde preferimos excepción controlada

Error al leer o escribir un fichero.

Fallo de conexión a una base de datos o red.

Acceso a un recurso externo (API, servicio web) que puede fallar.

Validación de datos obligatoria antes de procesar (por ejemplo, formato de archivo).

🔴 Situaciones donde preferimos excepción no controlada

Acceso a un objeto null.

División entre cero.

Pasar un índice inválido a un array o lista.

Violación de una regla interna de negocio que indica fallo de lógica (por ejemplo, saldo negativo que no debería ocurrir).

### 13. Pon un ejemplo en Java de firma de método que incluya throws, de una función que abre un fichero pero que declara que no le interesa menejar 
la excepción de si el fichero no existe, sino que se propague hacia arriba. Eso sí, acuérdate del finally.

Respuesta:

import java.io.*;

public class EjemploThrows {

    // Método que declara que puede lanzar IOException
    public static void abrirArchivo(String ruta) throws IOException {
        FileReader fr = null;

        try {
            fr = new FileReader(ruta);  // Puede lanzar FileNotFoundException (subclase de IOException)
            int c = fr.read();
            System.out.println("Primer carácter: " + (char)c);
        } 
        finally {
            // Se ejecuta siempre para liberar recursos
            if (fr != null) {
                fr.close();
                System.out.println("Fichero cerrado en finally");
            }
        }
    }

    public static void main(String[] args) {
        try {
            abrirArchivo("datos.txt");  // Llamador debe manejar la excepción
        } catch (IOException e) {
            System.out.println("Error propagado al main: " + e.getMessage());
        }
    }
}

### 15. ¿Cuándo se recomienda usar excepciones controladas, como IOException, y cuándo no controladas como IllegalArgumentException?
 ¿Existen en todos los lenguajes ambas opciones? En los que sólo existe una opción, ¿cuál es la más habitual?

 Respuesta:

 Cuándo usar excepciones controladas (checked)

Se usan para errores previsibles fuera del control del programador, generalmente relacionados con recursos externos o entornos imprevisibles.

Ejemplos típicos:

Lectura/escritura de ficheros: IOException

Conexiones de red: SocketException

Acceso a bases de datos: SQLException

Cuándo usar excepciones no controladas (unchecked)

Se usan para errores de programación, condiciones que no deberían ocurrir si el código es correcto.

Ejemplos típicos:

IllegalArgumentException → argumentos inválidos

NullPointerException → uso de objetos nulos

IndexOutOfBoundsException → índices fuera de rango

| Lenguaje   | Checked                | Unchecked | Comentario                                                                            |
| ---------- | ---------------------- | --------- | ------------------------------------------------------------------------------------- |
| **Java**   | ✅ Sí                   | ✅ Sí      | Java distingue explícitamente `checked` y `unchecked`.                                |
| **C#**     | ❌ No                   | ✅ Sí      | Solo hay excepciones no controladas; todas derivan de `Exception`.                    |
| **Python** | ❌ No                   | ✅ Sí      | Todas las excepciones son “unchecked” (se pueden capturar, pero no obligatoriamente). |
| **C++**    | ⚠ Opcional (con throw) | ✅ Sí      | C++ tiene `throw`, pero no fuerza al compilador a revisar.                            |
| **Go**     | ❌ No                   | ✅ Sí      | Go no tiene excepciones; usa `error` como valor de retorno.                           |


### 16. ¿Tiene sentido lanzar excepciones dentro del catch? ¿Se puede relanzar la misma excepción capturada? 
¿Cuándo tendría sentido hacer esto último? Pon ejemplos de ambos casos.

Respuesta:

| Caso                     | Cómo se hace                 | Cuándo tiene sentido                                                              |
| ------------------------ | ---------------------------- | --------------------------------------------------------------------------------- |
| Nueva excepción          | `throw new MiExcepcion(...)` | Cambiar contexto, tipar la excepción al dominio de la app                         |
| Relanzar misma excepción | `throw e;`                   | Hacer acciones adicionales (logging, metrics, limpieza) sin perder la propagación |


import java.io.*;

public class EjemploLanzar {
    public static void leerArchivo(String ruta) throws MiExcepcionAplicacion {
        try {
            FileReader fr = new FileReader(ruta);
        } catch (IOException e) {
            // Convertimos IOException en una excepción de nuestra aplicación
            throw new MiExcepcionAplicacion("Error al leer el archivo: " + ruta, e);
        }
    }
}

// Excepción personalizada
class MiExcepcionAplicacion extends Exception {
    public MiExcepcionAplicacion(String msg, Throwable causa) {
        super(msg, causa);
    }
}
